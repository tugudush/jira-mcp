# AGENTS.md — jira-mcp contributor guidance

This file follows the [AGENTS.md](https://agents.md) vendor-neutral convention
and is read by AI coding agents working on `tugudush/jira-mcp` (GitHub Copilot,
Cursor, Aider, Cline, Continue, Zed, etc.). Keep it terse and load-bearing —
every line should save a round-trip.

## Project in one paragraph

`jira-mcp` is a **read-only-by-default MCP server** that talks to **Jira Cloud
REST API v3** using Basic Auth (`email:api_token`). It ships **36 tools** (35
read + 1 scoped `jira_update_issue_text`). TypeScript 6 + ESM + Vitest 4 + ESLint
10 flat config + Prettier 3.8 + Zod 4. Architecture = domain handlers in
`src/handlers/` registered in `src/index.ts` via the modern `McpServer.registerTool`
API. Reference plans live in [`docs/plan.md`](docs/plan.md).

## Commands (run from `c:\htdocs\jira-mcp`)

| Command                           | What                                                 |
| --------------------------------- | ---------------------------------------------------- |
| `npm install`                     | install deps                                         |
| `npm run dev`                     | watch-mode dev runner (tsx)                          |
| `npm run build`                   | prebuild (sync version) + tsc → `dist/`              |
| `npm test`                        | single-run Vitest                                    |
| `npm run test:watch`              | Vitest watch                                         |
| `npm run test:coverage`           | Vitest + v8 coverage                                 |
| `npm run lint`                    | ESLint (max-warnings 0)                              |
| `npm run type-check`              | `tsc --noEmit`                                       |
| `npm run format` / `format:check` | Prettier write / check                               |
| `npm run ltfb`                    | lint → type-check → format → build (full local gate) |

Always finish work with `npm run ltfb && npm test`.

## Code patterns (must follow)

- **Handlers are one tool per file category** in `src/handlers/<category>.ts`.
  Pattern: `export async function handleFoo(args: z.infer<typeof FooSchema>): Promise<CallToolResult>`.
- **Schemas live in `src/schemas.ts`** and are wrapped with `withOutputOptions(...)`
  to add `output_format` + `filter` automatically.
- **API calls** use `makeRequest<T>()` (reads) or `makeIssueUpdateRequest<T>()`
  (scoped writes only). Never call `fetch` directly from a handler.
- **Read tools** stay free; **writes** go through `makeIssueUpdateRequest` which
  is method-locked to `PUT` and path-locked to `/^/rest/api/3/issue/[^/]+$/`.
- **Identity checks** (reporter/assignee) belong in the handler, not the API layer.
- **Tests** live in `tests/handlers/<category>.test.ts` and use `vi.mock('../../src/api.js', ...)`.
  Mirror the handler's exported functions.
- **No `any`.** Use `unknown` + Zod parsing when shape is dynamic.
- **ESLint complexity ≤ 10.** Extract deep-property accessors into helpers if a
  formatter hits the limit (e.g. `getFieldProp`, `getCategoryLine`).

## Environment & secrets

- All Jira secrets (`JIRA_BASE_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN`,
  `JIRA_ALLOW_ISSUE_UPDATES`, `JIRA_DEBUG`) come from the MCP client's `env` block
  (`.vscode/mcp.json` locally, or the IDE's MCP settings in CI/dev).
- The repo has **no `.env` file in git** — secrets stay in `.vscode/mcp.json`
  (user-global gitignored) or in the IDE's MCP store.
- `JIRA_DEBUG=true` logs URLs + status codes only — **never** log request bodies,
  response bodies, or headers (no `Authorization`).
- `.cjs` files (e.g. `scripts/dump-mcp-env.cjs`) are **prettier-only** in lint-staged,
  not linted by ESLint.

  index.ts                  # McpServer bootstrap + registerTool calls

## PR conventions

- One feature per PR, branch from `main`: `feature/phase-XX-<short-slug>`.
- PR title: `feat: <one-line summary>` (Conventional Commits).
- PR body must include:
  - "What landed" — files added/changed with a 1-line summary each
  - "Verification" — output of `npm run ltfb && npm test`
  - "Drift from plan §N" — anything that deviated from the plan, even small things
  - "Lessons learned" — gotchas future contributors should know
- All PRs must pass `npm run ltfb && npm test` locally on Node 20 and Node 22 before merge. There is no automated CI pipeline — the maintainer runs the gate locally.
